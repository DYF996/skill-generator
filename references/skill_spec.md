# Skill 生成规范

> 本文件是 `skill-generator` 的 Skill 生成模块。详细规定了需求理解、结构决策、SKILL.md 编写规范、references/scripts/assets 生成规则。详见 `interactive_protocol.md` 获取交互式求解的通用协议（风险分级、回滚、模式切换等）。

---

## 步骤 1：理解 Skill 需求

通过提问和推断，提取以下七要素：

| 要素 | 说明 |
|------|------|
| **任务目标** | 用户想自动化什么？一句话说清楚 |
| **触发场景** | 什么样的用户输入激活这个 Skill？包括具体关键词 |
| **输入** | 用户提供什么数据/参数？格式要求？ |
| **输出** | Skill 产出什么结果？格式要求？ |
| **关键步骤** | 完成这个任务需要哪几步？（≤10步） |
| **约束条件** | 有什么限制、边界、注意事项？ |
| **使用频率** | 一次性还是长期重复？ |

**信息不足时必须反问，不允许凭空假设。**

---

## 步骤 2：判断 Skill 结构

| 复杂度 | 判断标准 | 目录结构 |
|--------|----------|----------|
| **简单** | 步骤 ≤ 5 步；无外部依赖；无复杂参考 | 纯 SKILL.md |
| **中等** | 有 API/数据格式；多可选路径；有规则需记忆 | SKILL.md + references/ |
| **复杂** | 确定性重复执行；需模板文件；需自动化校验 | SKILL.md + references/ + scripts/ + assets/ |

---

## 步骤 3：递归防护检查

生成前检查目标 Skill 名称：

- **禁止名称**：`skill-generator`（大小写不敏感）
- **禁止模式**：尝试生成同名 Skill 时，报错退出并提示："skill-generator 不可生成，若要改进自身请直接修改自身文件"
- **改进自身**：用户要求改进 skill-generator 时，直接修改 `~/.qclaw/skills/skill-generator/` 下的对应文件，不调用 init_skill.py

---

## 步骤 4：编写 SKILL.md

### Frontmatter（YAML 头）

```yaml
---
name: <kebab-case-名称>
description: <描述：做什么 + 何时触发（含具体关键词）+ 生成内容清单>
---
```

**description 编写规范：**
- 第一句：概括 Skill 核心功能（≤30字）
- 第二句起：详细说明触发场景，包括用户可能说的话（关键词）
- 结尾：说明生成的输出内容（SKILL.md 必须、references/scripts/assets 按需）

**禁止在 description 中出现**：`<` 或 `>` 符号（YAML 规范）

### Body（正文）

必须包含以下章节：

```markdown
# <Skill 名称>

<概述：1-2句话说明核心功能>

## 执行步骤

1. <步骤一>
2. <步骤二>
3. <步骤三>

## 输入输出规范

- **输入**：<用户提供的参数或数据>
- **输出**：<Skill 产出的结果>

## 约束条件

- <约束一>
- <约束二>

## Few-Shot 示例

### 示例一：<描述>
- **输入**：<用户说>
- **输出**：<Claude 做什么>
```

**可选章节**（按需添加）：
- `## 前置条件`：使用 Skill 前需要具备什么
- `## 工具说明`：涉及哪些工具及其使用要点
- `## 进阶用法`：高级用户可选的额外功能

---

## 步骤 5：生成 references/（按需）

创建 Markdown 参考文档，命名规范：`references/<主题>.md`

适用场景：
- API 规范文档 → `references/api_spec.md`
- 数据格式定义 → `references/data_format.md`
- 业务规则说明 → `references/rules.md`
- 品牌/风格指南 → `references/guidelines.md`

**大文件规范（>100行）：** 文件顶部加目录（## 目录），便于选择性加载。

---

## 步骤 6：生成 scripts/（按需）

适用场景：
- 需要确定性重复执行的代码
- 反复重写的代码片段（>3次出现）
- 格式转换、数据校验逻辑

**测试要求**：必须实际运行测试，确保无语法错误、输出符合预期。至少测试一个核心用例。

**脚本命名规范**：`scripts/<动作>_<目标>.<ext>`
- 示例：`rotate_pdf.py`、`validate_csv.py`、`fetch_weather.sh`

**依赖说明**：在脚本文件顶部注释中写明依赖，如：
```python
#!/usr/bin/env python3
#依赖：pip install pandas openpyxl
```

---

## 步骤 7：生成 assets/（按需）

适用场景：
- 模板文件（PPT、Docx）→ `assets/template.pptx`
- 品牌资产（Logo、字体）→ `assets/logo.png`
- 前端模板 → `assets/frontend-template/`
- 示例数据 → `assets/sample_data.csv`

**assets/ 不加载到上下文**，仅在输出中引用或复制。

---

## 步骤 8：创建 Skill（文件系统操作）

**必须使用 init_skill.py**，禁止手动 mkdir + write：

```bash
# Linux/macOS
python scripts/init_skill.py <skill-name> --path ~/.qclaw/skills

# Windows
python scripts\init_skill.py <skill-name> --path %USERPROFILE%\.qclaw\skills
```

---

## 步骤 9：打包验证（可选）

```bash
python scripts/package_skill.py <skill-path>
```

验证通过后输出 `.skill` 包文件。

---

## 约束条件汇总

1. **最小可用**：SKILL.md 必填，至少 1 个 Few-Shot 示例
2. **禁止伪代码**：所有代码可执行，非示例代码需测试
3. **需求不清则反问**：宁可多问一句，不产四不像
4. **递归防护**：禁止生成 `skill-generator`
5. **文件创建**：必须用 init_skill.py

## 完整示例：周报生成 Skill

### 用户需求

> "每周写周报，能不能做个 skill？"

### 步骤 1：七要素提取

| 要素 | 内容 |
|------|------|
| 任务目标 | 将零散工作事项整理成结构化周报 |
| 触发场景 | 用户说"写周报"、"生成周报"、"本周总结" |
| 输入 | 本周完成事项 + 下周计划（可选项目名） |
| 输出 | Markdown 格式周报，含日期 |
| 关键步骤 | 收集信息→整理结构→量化补充→输出 |
| 约束条件 | 不虚构数据；下周计划缺失时提示 |
| 使用频率 | 每周重复 |

### 步骤 2：结构判断

→ 简单 Skill，纯 SKILL.md

### 步骤 3：递归检查

→ OK，未冲突

### 步骤 4：输出 SKILL.md

```markdown
---
name: weekly-report
description: 自动生成结构化周报。触发场景：用户说"写周报"、"生成周报"、"本周总结"、提供本周工作事项并要求整理成周报。输入：本周完成事项（必填）、下周计划（可选）；输出：Markdown 格式周报文档，含日期。
---

# 周报生成器

根据用户提供的本周工作内容和下周计划，自动生成结构化周报。

## 执行步骤

1. **收集信息**：确认本周完成事项、下周计划、团队/项目名（若缺失，反问）
2. **整理结构**：按"本周总结 → 本周亮点 → 下周计划 → 需要支持"四段式组织
3. **量化补充**：将事项尽量量化（完成率、工时、影响范围）
4. **输出周报**：Markdown 格式，标题含周次和日期

## 输入输出规范

- **输入**：本周完成事项（列表）、下周计划（列表）、可选（团队名、项目名）
- **输出**：Markdown 格式周报，含周次日期

## 约束条件

- 若未提供下周计划，主动提示："下周计划还没填，需要补全吗？"
- 不虚构未提供的数据
- 周报语言与用户输入一致

## Few-Shot 示例

### 示例一：完整输入
- **输入**："帮我写周报。本周：完成登录模块重构、推动与XX部门对齐。下周：开始订单模块、准备Q2复盘"
- **输出**：
  ```markdown
  ## 第13周周报（3/24-3/28）

  ### 本周总结
  - 完成登录模块代码重构，提交 MR 并合并
  - 推动与XX部门对齐会议，明确Q2协作计划

  ### 本周亮点
  - 登录失败率从 1.2% 降至 0.3%

  ### 下周计划
  - 开始订单模块开发（预计3天）
  - 准备Q2复盘材料

  ### 需要支持
  - 订单模块需XX系统API文档
  ```
```

### 示例二：信息不全
- **输入**："写周报，本周做了很多事"
- **输出**："好的！请提供具体事项，我来帮您整理。格式参考：① 完成了什么 ② 推进了什么 ③ 解决了什么问题"
```

### 步骤 8：创建文件

```bash
python scripts/init_skill.py weekly-report --path ~/.qclaw/skills
```

weekly-report/
